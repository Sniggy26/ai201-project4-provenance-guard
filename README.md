# Provenance Guard

A backend API that classifies submitted text as AI-generated or human-written, returns a confidence score, surfaces a transparency label, and handles creator appeals. Built with Flask, Groq (LLM signal), and stylometric heuristics.

---

## Architecture

A submitted piece of text takes the following path through the system:

1. `POST /submit` receives `text` and `creator_id`
2. **Signal 1 (LLM):** Groq's `llama-3.3-70b-versatile` assesses the text holistically and returns a score from 0.0 (human) to 1.0 (AI)
3. **Signal 2 (Stylometrics):** Pure Python heuristics compute sentence length variance, type-token ratio, and punctuation density — returning a score from 0.0 to 1.0
4. **Confidence scoring:** Weighted average of both signals (`0.6 × llm + 0.4 × stylo`)
5. **Transparency label:** Score mapped to one of three label variants
6. **Audit log:** Full entry written to `audit_log.json`
7. Response returned to caller with `content_id`, `attribution`, `confidence`, and `label`

For appeals:
1. `POST /appeal` receives `content_id` and `creator_reasoning`
2. Original log entry located by `content_id`
3. Status updated to `under_review`, reasoning appended
4. Confirmation returned to caller

## Detection Signals

### Signal 1: LLM Classification (Groq)
- **What it measures:** Semantic and stylistic coherence — unnatural fluency, overuse of transition phrases, lack of personal voice, uniform sentence structure
- **Why:** A large language model can assess writing holistically in ways rule-based systems can't
- **Output:** Float 0.0–1.0 (0.0 = human, 1.0 = AI)
- **What it misses:** Highly polished human writing (academic papers, professional blogs) shares surface features with AI output. Unreliable on very short texts.

### Signal 2: Stylometric Heuristics
- **What it measures:** Three statistical properties:
  - **Sentence length variance** — AI text tends to have consistent sentence lengths; human writing is more variable
  - **Type-token ratio (TTR)** — vocabulary diversity; AI text is more repetitive
  - **Punctuation density** — AI text clusters around moderate, predictable punctuation density
- **Why:** These properties are structurally independent from the LLM signal — one is semantic, one is statistical
- **Output:** Float 0.0–1.0 (0.0 = human, 1.0 = AI)
- **What it misses:** Non-native English speakers and academic writers may produce statistically uniform text that scores as AI-like purely due to writing style

### Combining Signals
confidence = (0.6 × llm_score) + (0.4 × stylo_score)
LLM signal weighted higher because it captures semantic patterns heuristics miss. Stylometric signal acts as a corrective — if LLM scores high but stylometrics score low (high variance, rich vocabulary), confidence is pulled toward uncertain.

## Confidence Scoring

| Score Range | Interpretation |
|-------------|----------------|
| 0.00 – 0.35 | Likely human |
| 0.36 – 0.64 | Uncertain |
| 0.65 – 1.00 | Likely AI |

Thresholds are intentionally asymmetric — the "likely AI" band doesn't start until 0.65 because a false positive (labeling a human's work as AI) is worse than a false negative on a creative writing platform.

### Example Submissions

**High-confidence AI text:**
Text: "Artificial intelligence represents a transformative paradigm shift in modern

society. It is important to note that while the benefits of AI are numerous, it is

equally essential to consider the ethical implications..."
llm_score: 0.8

stylo_score: 0.315

**Clearly human text:**

stylo_score: 0.177

confidence: 0.131

attribution: likely_human

The two scores differ by 0.475 — a meaningful separation across the confidence range.

## Rate Limiting

**Limits:** 10 requests per minute, 100 requests per day (per IP address)

**Reasoning:**
- A legitimate creator submitting their own work would rarely exceed 10 submissions in a minute — even bulk uploading a portfolio takes time between posts
- 100/day covers heavy legitimate use while making it expensive to flood the system programmatically
- These limits are applied per IP, so a single adversarial script can't bypass them by changing creator_id

## Appeals Workflow

**Endpoint:** `POST /appeal`

**Required fields:**
- `content_id` — from the original `/submit` response
- `creator_reasoning` — free text explanation from the creator

**What happens:**
1. System looks up the original classification by `content_id`
2. Status updated from `classified` to `under_review`
3. Appeal reasoning appended to the audit log entry
4. Confirmation returned to the creator

**Example request:**
```bash
curl -s -X POST http://localhost:5001/appeal \
  -H "Content-Type: application/json" \
  -d '{"content_id": "da18846c-afc1-4c2a-9385-fac992b8af91", "creator_reasoning": "I wrote this myself. I am an academic researcher and my writing style is formal by training, not because it is AI-generated."}' | python -m json.tool
```

**Example response:**
```json
{
  "content_id": "da18846c-afc1-4c2a-9385-fac992b8af91",
  "message": "Appeal received. Your submission is now under review.",
  "status": "under_review"
}
```

## Known Limitations

**1. Non-native English speakers:** Writers whose first language is not English often produce formal, grammatically uniform text that scores high on stylometric AI-likelihood — low sentence variance, moderate TTR — even when writing entirely by hand. The LLM signal may also flag this writing as AI-generated due to its structural regularity. These writers are the most likely to be falsely labeled. The appeals workflow exists specifically for this case, and the label text acknowledges that automated analysis is imperfect.

**2. Lightly edited AI output:** A user who generates text with AI and then edits it substantially may produce content that scores in the uncertain range. Neither signal can reliably detect AI text that has been meaningfully revised by a human. The system returns "uncertain" rather than a confident verdict in this case, which is the honest response.

## Spec Reflection

**One way the spec helped:** The requirement to write out all three transparency label variants in planning.md before building forced me to decide exactly what text a non-technical user would see at each confidence level. This made the label generation function trivial to implement — the spec section was essentially the function's logic written in plain English.

**One way implementation diverged:** The spec suggested the "likely AI" threshold would be a straightforward design decision. In practice, calibrating the threshold required testing on real inputs — the academic writing example scored 0.606 (uncertain) despite the LLM scoring it 0.8, because the stylometric signal correctly identified structural variation in the text. I kept the threshold at 0.65 rather than lowering it, because the system hedging toward uncertain on borderline cases is the right behavior for a platform where false positives are costly.

## AI Usage

**1. Flask app skeleton and Signal 1:** I gave Claude my planning.md detection signals section and architecture diagram and asked it to generate the Flask app skeleton with the POST /submit route and the Groq signal function. The generated function returned raw text instead of a float — I added explicit float parsing with a fallback to 0.5 for unparseable responses.

**2. Confidence scoring and Signal 2:** I gave Claude my uncertainty representation section and asked it to generate the stylometric heuristics function and weighted scoring logic. The generated scoring used equal weights for all three stylometric sub-signals — I revised the weights to 0.5/0.3/0.2 (variance/TTR/punctuation) after testing showed sentence length variance was the most reliable differentiator on my test inputs.
